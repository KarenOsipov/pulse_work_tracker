# ПУЛЬС — система мониторинга вовлечённости (мульти-устройство)

Одна админ-панель, к которой подключаются несколько сотрудников со своих
компьютеров. Живая статистика, текстовые отчёты и просмотр камеры "по запросу"
с явным уведомлением сотрудника. Работает как статический сайт (можно
разместить на GitHub Pages), а вся "серверная" часть — это бесплатный Firebase.

## Важно прочитать перед запуском

- **Согласие сотрудника обязательно** — страница сотрудника не начинает
  ничего записывать и передавать, пока человек не нажмёт «Я уведомлён(а) и
  согласен(на)».
- **Видно, когда админ смотрит камеру** — в момент просмотра у сотрудника
  на экране появляется баннер «Администратор сейчас смотрит вашу камеру».
  Скрытого наблюдения в этой системе нет и не должно быть.
- В разных странах/регионах свои требования к мониторингу сотрудников
  (уведомление в трудовом договоре, согласование с профсоюзом/советом
  работников, ограничения на видеозапись и т.д.). Перед реальным
  использованием стоит свериться с локальным трудовым законодательством —
  это не юридическая консультация, а просто напоминание.

## Как это устроено

- **GitHub Pages** отдаёт только статические файлы (HTML/JS/CSS) — своего
  сервера у него нет. Поэтому вся синхронизация данных между сотрудниками и
  админом идёт через **Firebase** (бесплатный тариф Spark более чем
  достаточен для небольшой команды):
  - **Firebase Authentication** — вход админа (email/пароль) и анонимный
    вход сотрудников (чтобы у каждого устройства был свой ID без пароля).
  - **Firestore** — база данных, где хранится статус и статистика каждого
    сотрудника и текстовые отчёты.
  - **WebRTC** — живое видео идёт напрямую между браузером сотрудника и
    браузером админа (peer-to-peer), Firestore используется только для
    "знакомства" браузеров друг с другом (обмен служебными сигналами).

## Шаг 1 — создать проект Firebase

1. Зайти на https://console.firebase.google.com → **Add project** →
   придумать имя.
2. **Build → Authentication → Get started**:
   - вкладка **Sign-in method** → включить **Email/Password**.
   - там же включить **Anonymous**.
3. **Build → Firestore Database → Create database** → выбрать
   **Production mode** → выбрать регион.
4. **Project settings (шестерёнка) → General → Your apps → Web (</>) ** →
   зарегистрировать приложение → скопировать объект `firebaseConfig`.
5. Вставить этот объект в файл `shared/firebase-config.js` вместо
   `"ВСТАВЬ_СЮДА"`.

## Шаг 2 — создать аккаунт администратора

Специально **нет** формы саморегистрации админа на сайте — иначе аккаунт
администратора смог бы создать кто угодно, кто просто откроет ссылку.

1. Firebase Console → **Authentication → Users → Add user**.
2. Указать свой email и пароль. Это и есть логин для `admin.html`.
3. Можно добавить несколько таких пользователей — все с ролью "не анонимный"
   автоматически получают права администратора (см. правила ниже).

## Шаг 3 — правила безопасности Firestore

Firebase Console → **Firestore Database → Rules** → вставить:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    match /employees/{employeeId} {
      // Сотрудник может писать только в свой собственный документ
      allow write: if request.auth != null && request.auth.uid == employeeId;

      // Читать список всех сотрудников может только "настоящий" (не анонимный)
      // авторизованный пользователь — то есть админ, вошедший по email/паролю
      allow read: if request.auth != null
                  && request.auth.token.firebase.sign_in_provider != 'anonymous';

      // Подколлекция звонков (WebRTC-сигналинг): доступна любому
      // авторизованному пользователю — это только служебные данные для
      // установки видеосвязи, само видео через Firestore не передаётся
      match /calls/{callId} {
        allow read, write: if request.auth != null;
      }
    }
  }
}
```

Это рабочая модель для небольшой команды. Для более строгой защиты в
будущем можно перейти на Custom Claims через Cloud Functions (проверка роли
"admin" на сервере), но это уже отдельная надстройка.

## Шаг 4 — запуск локально (проверить перед деплоем)

Открыть `index.html` через **локальный сервер**, не двойным кликом — камере и
модулям ES нужен `http://`, а не `file://`. Проще всего:

```bash
cd pulse-suite
python3 -m http.server 8000
```

Открыть `http://localhost:8000`.

## Шаг 5 — деплой на GitHub Pages

1. Создать репозиторий на GitHub, залить туда содержимое папки `pulse-suite`
   (включая `shared/firebase-config.js` с уже вставленными ключами).
2. **Settings → Pages → Build and deployment → Source: Deploy from a
   branch** → выбрать branch `main`, папку `/ (root)` → Save.
3. Через минуту-две сайт появится по адресу вида
   `https://<твой-логин>.github.io/<репозиторий>/`.
4. Ссылку на `employee.html` дать сотрудникам, ссылку на `admin.html`
   оставить только себе.

> Repo можно сделать публичным — секретных данных в `firebase-config.js` нет
> (это публичный клиентский ключ), реальная защита — это правила Firestore
> из шага 3. Если всё же хочется приватности кода, GitHub Pages также
> работает и с приватными репозиториями на платных тарифах GitHub.

## Если видео между админом и сотрудником не подключается

Публичные STUN-серверы (уже прописаны в `firebase-config.js`) отлично
работают в большинстве офисных и домашних сетей. Если оба участника за
жёстким корпоративным firewall/NAT — WebRTC может не установить прямое
соединение. В этом случае нужен **TURN-сервер** (ретранслятор трафика).
Бесплатный вариант на пробу — https://www.metered.ca/tools/openrelay/,
добавить выданные там серверы в массив `iceServers` в
`shared/firebase-config.js`.

## Структура проекта

```
index.html          # стартовая страница — выбор "сотрудник" / "админ"
employee.html         # рабочее место сотрудника: согласие, ИИ-анализ, синхронизация
admin.html             # панель администратора: список сотрудников, статистика, живое видео
shared/firebase-config.js  # ключи Firebase + список ICE-серверов для WebRTC
```

## Ограничения, о которых честно стоит знать

- Оценка "настроения" по мимике — эвристика с известными ограничениями
  точности, не медицинская/психологическая диагностика.
- Детекция "разговора" и "телефона в кадре" — тоже эвристики, возможны
  ложные срабатывания.
- Полный посекундный таймлайн синхронизируется только локально у
  сотрудника; в админ-панель уходят агрегированная статистика и текстовые
  сводки каждые N минут — так база данных не перегружается лишними
  записями.
